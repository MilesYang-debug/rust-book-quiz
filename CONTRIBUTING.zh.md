# 参与贡献 Rust Book Quiz

[English](CONTRIBUTING.md) | 中文

感谢你帮助改进题库！Issue 和 PR 用中文或英文都欢迎。

最有价值的两类贡献是**报告有问题的题目**（答案错误、选项有歧义）和**新增题目**。
两者走同一套机制：`bank/` 下的所有改动在每个 PR 上都会被 CI 自动校验。

## 报告有问题的题目

开一个 Issue，写上题目 `id`（JSON 里的 `chNN-qNN`；应用内的章节和 section 标签
能帮你定位）。说明你认为哪里有问题，最好引用 The Book 中能一锤定音的原文——
每道题答完后都有"查看原文"链接，出处一点即达。

## 新增题目

JSON 格式、文件布局和同步流程见
[README.zh.md → 题库](README.zh.md#题库自己加题--贡献题目)。简版流程：

```bash
# 1. 编辑 bank/chNN.json
# 2. 校验 + 重新生成内嵌快照（两者都要提交！）
cd app && cargo run -p quiz-bank -- --sync
# 3. 用真实 rustc 机械验证代码片段的行为断言
cargo run -p quiz-bank --bin code_check
```

CI 在你的 PR 上跑的就是这几条（`.github/workflows/validate.yml`），本地过了
CI 就能过。

### CI 会自动检查的

- Schema：`id` 唯一、`tag` 在合法集合内、`difficulty` 1–3、选项键 `A`–`F`
  （至少 3 个；多选题需要 5 个）、`answer` 字母必须存在、文本非空。
- `section` 必须写成 `"N.M Title"` 并能映射到 The Book 的真实章节（章号要和
  文件一致）。应用靠这个字段自动生成每道题的"查看原文"链接——不需要手工维护
  URL。目录快照在 `app/quiz-bank/src/book_toc.rs`。
- `anchor`（可选）必须是小写连字符的 URL 锚点；填了之后"查看原文"会直接跳到
  该小节页面里的具体标题。
- `app/assets/bank.json` 必须同步（忘了跑 `--sync` 会挂 CI）。
- 带 `code` 字段的题目，`code_check` 会用本地 `rustc` 实际编译（并运行）代码，
  和正确选项里的断言交叉核对："编译不过（E0382）""会 panic""测试失败"这类
  说法都会被真实验证。

### 需要人工把关的：题目质量

自动校验分不出好题和坏题。提交前请逐题过一遍这几条：

1. **有且仅有一个站得住脚的答案。** 题库最致命的失败模式，就是标准答案能被
   仔细的读者合理质疑。如果某个干扰项"其实也说得通"，就把它改到严格错误为止。
2. **干扰项应是真实存在的误解**，不是凑数的。最好的错误选项是人们真的会相信
   的东西（`Misconception` 标签就是为此设的）。
3. **解析要能教会人。** 解析是写给刚选错的人看的——要回答"我为什么会选错"，
   而不是只确认正确答案。既说清为什么对，也说清最诱人的干扰项为什么错。
4. **代码片段自成一体。** 应当在没有隐藏上下文的情况下，按题目声称的方式编译
   （或编译失败）；否则 `code_check` 会指出来。
5. **锚定原书。** `section` 字段是"The Book 讲过这个"的承诺——不要考民间
   传说或不稳定特性。

## 其他贡献

应用代码（Leptos 前端、Tauri 壳）、CI、文档——正常 PR 流程，没有特殊规则。
改动较大时请先开 Issue 讨论设计，再投入时间。
